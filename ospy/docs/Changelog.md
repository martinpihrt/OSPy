July 22 2026 (v3.0)
-----------
(martinpihrt)<br/>
Moved the three experimental SQLite controls from the general System section into a dedicated collapsible **Settings storage** card in Options. The controls remain separate during beta validation so emergency recovery, verified reads and strict dual-write commits can each be disabled independently without changing stored data. Updated all seven Web Interface Guides and added a regression test that keeps only these three controls in the dedicated category.

Added persistent, non-authoritative SQLite migration evidence. OSPy now atomically records consecutive successful starts through the verified SQLite read path and consecutive successful strict dual-write commits; a failure resets only the affected streak and preserves its time and reason. Diagnostics displays both streaks and the last failure. The evidence file contains no settings and cannot influence startup, recovery or backend selection. Added atomic persistence, corrupt-status fallback and integration regression tests and updated all seven Web Interface Guides.

Prepared the early interface-language bootstrap for a future SQLite-primary mode without changing the current authority. It still opens shelve/DBM first and, only when verified SQLite reads are enabled, verifies the complete SQLite key and checksum set against that exact snapshot before decoding the single `lang` value. Any missing, legacy, divergent, corrupt or invalid SQLite copy falls back to the shelve language. Missing settings databases are now opened read-only and can no longer create an empty shelve file during the language probe. Added verified-read, corruption-fallback, disabled-path and no-file-creation tests and updated all seven Web Interface Guides.

Localized the displayed choices for the web theme, station-image source and back-to-top position. Their stable English internal values remain unchanged in the settings database for full compatibility across language changes and existing installations. Added a regression test for the value/label separation and updated all seven Web Interface Guides.

July 21 2026 (v3.0)
-----------
(martinpihrt)<br/>
Introduced the first non-migrating settings-storage transition phase. OSPy still reads and writes the existing shelve/DBM settings files through a new storage interface; no SQLite settings file is created and no user data are converted. Diagnostics now passively reports the active backend and Python SQLite readiness without treating unavailable SQLite as a system failure. The clean installer verifies built-in SQLite support with an in-memory integrity check, and automated tests cover shelve compatibility, absence of SQLite filesystem writes and safe fallback when SQLite is unavailable. Updated the clean-install guide and all seven Web Interface Guides.

Added the second settings-storage transition phase: every successful shelve/DBM save now produces a non-authoritative SQLite shadow copy through a temporary file. OSPy atomically replaces it only after schema, key, serialized-value and SQLite integrity verification, and never reads it during startup. Shadow failure cannot block or invalidate the authoritative shelve save; Diagnostics reports synchronized, pending, unavailable or failed shadow state and keeps a shadow-only failure at warning severity. Added round-trip, corruption, atomic cleanup and forced-failure regression tests and updated all storage documentation.

Corrected the in-memory last-settings-save timestamp after an atomic database-directory replacement. Diagnostics now immediately shows the same successful save time as the SQLite shadow and database heartbeat, while a failed primary replacement leaves the previous timestamp unchanged.

Added the third settings-storage transition phase. SQLite shadow schema 2 stores a SHA-256 checksum beside every serialized value, verifies those checksums on disk and compares them with the authoritative shelve values at OSPy startup. The automatic comparison never unpickles shadow data and never uses SQLite to recover or start OSPy. Diagnostics distinguishes verified, diverged, damaged, pending and schema-upgrade states; every mismatch remains a warning while shelve stays authoritative. Existing schema 1 shadows are left unread and atomically replaced on the next successful settings save. Added matching, divergence, checksum-tampering, legacy-schema and authoritative-startup regression coverage and updated all seven Web Interface Guides.

Added the fourth settings-storage transition phase. After every key and SHA-256 checksum matches the authoritative shelve snapshot, OSPy performs a read-only dry-run decode and applies the normal settings validator to the reconstructed SQLite dictionary. The reconstructed values are discarded and never assigned to the running configuration; any decode or validation failure remains a warning while startup continues with shelve. Diagnostics now reports the SQLite read-test result, and tests prove both successful reconstruction and that checksum mismatch prevents every `pickle.loads` call. Updated all seven Web Interface Guides.

Run the guarded SQLite read test immediately after every successful atomic shadow replacement as well as during startup. This prevents a routine startup settings save from replacing a passed read-test result with a permanent “waiting for next start” state. A post-save read-test failure still leaves the primary shelve save valid and is covered by a dedicated regression test.

Added the fifth settings-storage transition phase. SQLite shadow schema 3 adds an independently verifiable manifest containing the exact record count and a checksum over the complete sorted key/checksum list, so removed or inserted rows are detected before decoding. OSPy now performs and discards full recovery dry runs for both the current shadow and the previous shadow in the backup directory, applies the normal settings validator and reports both results in Diagnostics. Recovery candidates are still never selected for startup; failure leaves shelve/DBM authoritative. Added current/backup recovery, deleted-row, pre-unpickle rejection and forced-recovery-failure tests and updated all seven Web Interface Guides.

Added the sixth settings-storage transition phase. At startup OSPy selects the first independently verified SQLite recovery candidate only for an isolated restore rehearsal, writes it into a temporary shelve/DBM directory, reopens and validates the result, compares every serialized setting and then removes the temporary directory. The active database and startup source are never modified; if the current SQLite copy is damaged, the rehearsal may test the verified backup while live settings still come from authoritative shelve. Diagnostics reports the candidate, record count and result, and any failure remains a warning. Added successful, backup-fallback and forced-failure regression tests and updated all seven Web Interface Guides.

Preserved the startup SQLite restore-rehearsal result across later routine settings saves, so Diagnostics continues to show the completed rehearsal instead of silently losing its status when the shadow-copy state is refreshed.

Added the seventh settings-storage transition phase: a read-only emergency-selection dry run records which independently verified SQLite candidate would be chosen if every shelve/DBM startup candidate were invalid. The current copy has priority and the verified backup is selected only when the current copy fails. The selected dictionary is deliberately discarded and never assigned to running options, including in the all-shelve-invalid test scenario. Diagnostics reports readiness, hypothetical source and record count, and routine saves preserve the startup result. Added current-priority, backup-fallback, all-shelve-invalid and status-retention tests and updated all seven Web Interface Guides.

Added the eighth settings-storage transition phase with explicitly opt-in emergency recovery, disabled by default. Enabling it in System options creates a non-secret atomic marker only after a schema-3 SQLite copy passes independent recovery verification; the verified SQLite dictionary must also contain the enabled option, so neither the marker nor the database can authorize recovery alone. Only when the normal current, temporary and backup shelve/DBM candidates are all invalid does OSPy rebuild and revalidate a shelve database in the temporary settings directory, preferring current SQLite and falling back to its verified backup. Existing current and backup directories are not overwritten by the recovery step, and normal persistence remains shelve/DBM. Disabling removes the marker before any database write and therefore fails closed even if that write fails. Diagnostics and startup recovery messages identify actual use or failure. Added default-off, dual-authorization, current/backup recovery, atomic marker and forced-write-failure tests and updated all seven Web Interface Guides.

Added the ninth settings-storage transition phase with an experimental verified SQLite read path, disabled by default. Even when enabled, OSPy first loads and validates shelve/DBM, compares the complete SQLite key and checksum set against that exact authoritative snapshot, independently decodes and validates every value, and only then assigns the equivalent reconstructed dictionary to the running options. Any missing copy, checksum difference, database error or decode/validation failure keeps the already loaded shelve values without interruption. Writes, atomic directory rotation and backups remain exclusively shelve/DBM with SQLite as the verified shadow. Diagnostics reports used, disabled or shelve-fallback state and preserves the startup result through routine saves. Added opt-in success, default-off, divergence, decode-failure and status-retention tests and updated all seven Web Interface Guides.

Added the tenth settings-storage transition phase with an experimental strict dual-write commit policy, disabled by default. When enabled, a settings transaction is allowed to replace the active directory only after both the temporary shelve/DBM database and its SQLite shadow have been written and independently verified. If SQLite is unavailable or its write or verification fails, OSPy removes the entire uncommitted temporary directory, leaves the previous active settings and last-save timestamp unchanged, and reports the normal settings-save problem. Disabling the option restores compatible shelve-first behavior. Successful strict commits remain readable through the verified SQLite read path, while shelve/DBM continues to be the persistence and backup format. Diagnostics reports the active commit policy. Added successful round-trip, forced SQLite failure, uncommitted-directory cleanup, reload and default-off tests and updated all seven Web Interface Guides.

Updated the Czech translation catalog with the complete SQLite storage readiness, shadow verification, guarded read-test and recovery dry-run status messages.

Added a shared weather-provider layer for OSPy and weather-aware plug-ins. Options now offer the key-free Open-Meteo automatic model, CHMI ALADIN through Open-Meteo, and the existing Stormglass service. Legacy installations with a saved Stormglass key retain Stormglass while keyless installations migrate to Open-Meteo; invalid provider values are rejected, cached data remain available during temporary failures, and providers never change silently.

Added a non-blocking three-card forecast to the Home weather section for the current time and approximately three and six hours ahead. It uses local SVG weather icons, shows temperature and precipitation, identifies the selected data source and last cache update, and is styled responsively in all three themes. Corrected the fallback evapotranspiration calculation to use atmospheric pressure instead of wind speed.

Added automated provider, migration, normalization, rendering and calculation coverage. Updated all seven Web Interface Guides with provider selection, compatibility, API-key, attribution, cache and Home forecast behavior.

Added revision-based cache invalidation to the active theme stylesheet and explicit intrinsic dimensions to Home forecast icons. After an OSPy update, browsers now load the matching CSS immediately and cannot render newly introduced SVG icons at their full document size while an older stylesheet is cached.

Hardened and simplified the clean Raspberry Pi OS/Debian installation path. The installer now requires Python 3.11 or newer, clones only the stable `master` branch, preserves existing checkouts, installs core dependencies from Debian packages, keeps MQTT and multimedia support optional, and no longer downloads or executes separate Astral, MySQL Connector or log2ram archives. It renders the versioned native systemd template with absolute paths, starts OSPy, verifies service health and prints recent service output on failure. Cancelling or deferring a reboot no longer reports a successful installation as failed. Updated the clean-install guide, service instructions and all seven Web Interface Guides, and added installer syntax, safety, dependency, service and documentation regression tests.

Updated the Czech translation catalog with the weather-provider selection, short-term Home forecast and related diagnostics messages.

Expanded GitHub Actions compatibility testing from the Raspberry Pi OS Python 3.11 baseline to a required Python 3.11 and latest stable Python 3.14 matrix. Both jobs run the same complete OSPy and official plug-in suite, fail independently and must succeed; the clean installer now warns only for versions newer than the tested range.

Fixed the local web.py compatibility fallback on Python 3.13 and newer, where the removed standard-library `cgi` module exposed a circular package import while loading `cgi_compat`. The web API now imports only the required `FieldStorage` class, preserves the original Python 3.11 path and includes a regression test that simulates a Python installation without `cgi`.

Fixed web.py template parsing on Python 3.14. The stricter tokenizer no longer rejects valid OSPy or plug-in templates when translated text or HTML immediately following a template expression contains an unmatched quote, and a trailing punctuation dot is no longer mistaken for attribute access. Added regression coverage and verified every OSPy and official plug-in template on Python 3.14 while retaining the Python 3.11 behavior.

Removed the remaining Python 3.14 invalid-escape `SyntaxWarning` messages from Wi-Fi signal parsing, the web.py debug page and bundled Blinker documentation strings without changing their runtime behavior. The complete Python source tree now compiles with `SyntaxWarning` promoted to an error on Python 3.14.

Reworked system backup and restoration. New backups contain a versioned manifest with OSPy metadata, file sizes and SHA-256 checksums and include settings, system event data and station images while excluding SSL private keys, plug-in code and Python caches. Uploaded archives are bounded and checked for unsafe paths, links, encryption, duplicates, suspicious compression, undeclared files and checksum mismatches before staging. OSPy creates a pre-restore safety backup and atomically swaps verified data, restoring the previous directories if the swap fails; legacy data-only backups remain supported under the same ZIP safety checks. The ten newest on-device safety backups are retained and listed in Options for download; the interface warns that system archives contain credentials and must be protected. Improved manual plug-in updates. Plug-in names remain white and readable on the yellow animated update highlight. An installed plug-in can now be manually updated from Plug-in management, its Changes page or Install new plug-in even when the available version currently reports a compatibility error, allowing an update to repair an older broken installation; an incompatible updated version is left disabled, while a fresh incompatible installation and an automatic incompatible update remain blocked. Added regression coverage for the distinction.

Fixed sensor loading after strict persistent-data validation exposed historic representations of the configured IP address and regulation output. Stored IPv4 octets and numeric strings are now migrated to validated canonical values. A configured sensor is identified by its stable MAC address and a valid IP reported by its packet automatically restores a lost address or follows a DHCP change; invalid addresses are rejected. New form values remain numeric, and both the sensor list and edit templates use the same representation. External-IP discovery now refreshes its cache in a background worker instead of blocking the scheduler and web requests for up to two network timeouts; this prevents healthy schedule calculation from being reported as unresponsive while an IP service is slow. Added regression tests for the migrations, MAC-based packet recovery, Sensors page rendering and non-blocking network refresh.

Hardened startup against damaged persistent data. Settings candidates are now opened read-only, always closed after validation, checked for core value types and critical startup bounds, and rejected in favor of the temporary or backup database when invalid. Invalid fields in stored program, station, sensor and other object records are ignored individually while safe defaults remain active; stale unknown fields are not restored. Malformed watering, e-mail and event log entries are discarded before rendering or scheduler use. The disposable web-session database removes unreadable entries or is recreated if its storage is damaged, and it now follows the isolated data directory used by automated tests. Settings recovery is recorded as a warning in the system event log. Added isolated recovery tests without modifying production data.

Added a coordinated, bounded shutdown lifecycle for OSPy. Scheduler, sensor, weather and plug-in repository workers now use cooperative stop events instead of unconditional infinite sleeps, all running plug-ins receive their existing stop callback within one shared shutdown window, and pending settings are written synchronously before databases close. Restart and shutdown stop accepting web work, prevent new scheduler activity, clear active runs and force station and master-relay outputs off both before and after plug-in callbacks. Threads that do not stop within the limit are logged without blocking process restart indefinitely. Added isolated tests for interruptible workers, output safety, shutdown ordering, idempotence and pending-settings persistence. Background weather activity is now disabled in the isolated test environment, and options cleanup drains an already running delayed writer before its temporary data directory is removed.

Added isolated sensor and weather recovery tests. The suite now verifies sensor offline timeout handling, recovery from a valid packet, malformed-packet isolation, weather location retries, response validation and reuse of the last valid cached weather value after a refresh failure. Weather location and Stormglass requests now have a bounded 15-second timeout, transient location failures preserve the last valid coordinates, failed location lookups are retried, and empty or explicit Stormglass error payloads are not cached as valid data. The tests do not contact sensors or Internet services. Updated the built-in automated test guide.

Expanded automated output and station safety coverage with an in-memory shift-register driver. Tests now verify activation, deactivation, clearing all outputs, sequential and parallel schedules, primary and secondary master relays, master on/off delays, immediate stop-all behavior and scheduler recovery after an output exception. Hardened station control so negative or out-of-range identifiers never address an output, disabled stations and master outputs cannot be started manually from Home or the API, and Stop All immediately disables the separate master relay as well as clearing automatic run state. Invalid API station starts now return `400 Bad Request`; the API documentation and built-in test guide describe the behavior.

Added isolated web-route integration tests that render the main administrator, plug-in management and plug-in installation pages through the real web.py dispatcher. The tests also cover the anonymous login page and redirect, administrator-only Diagnostics JSON, successful and failed Diagnostics responses, no-cache headers, unknown plug-in history, role rejection and CSRF enforcement before a state-changing handler runs. Device temperature, CPU, network, uptime and plug-in repository values are deterministic during rendering, so the tests do not access host hardware or external services. Updated the built-in automated test guide.

Prevented transient plug-in initialization states from opening a false Diagnostics error dialog on the Home page after an OSPy restart or update. A plug-in `health()` error now has to be reported by two consecutive Home checks, while a failed start, runtime exception, incompatible manifest or failed pre-activation test is still reported immediately. The Diagnostics page continues to show every current state without delay. Added regression tests for the failure classification.

Hardened plug-in ZIP installation before extraction. OSPy now rejects unsafe or non-portable paths, Unicode and case-insensitive duplicates, duplicate plug-in identifiers, symbolic links, special or encrypted entries, damaged archives, excessive compression ratios and bounded file, archive and expanded-size limits. Each plug-in is staged on the destination filesystem and installed atomically while preserving its `data` directory; a failed replacement or start restores the previous files and plug-in status. Added automated tests for rejected archives, no-write validation, data preservation and rollback, and documented the accepted ZIP structure and limits in all seven Web Interface Guides and the built-in plug-in developer and test guides.

Reduced transient HTTP 408 errors during Diagnostics refresh. Overlapping browser requests are suppressed, plug-in health checks are collected concurrently within one shared timeout, and their result is briefly cached for simultaneous diagnostics endpoints. A single transient timeout remains visible as an inline refresh status but no longer opens a persistent error dialog.

Added the optional **Diagnostics** setting for showing system-health errors to administrators on the Home page. The Home page uses the same red problem, details and possible-solution dialog as Diagnostics, polls without overlapping requests, and gives a health error priority over the existing update notification. Dismissing one notification no longer hides a different higher-priority notification. Added matching styles for all built-in themes and documented the option in all seven Web Interface Guides.

Expanded the isolated automated test suite with scheduler coverage for rain delay, manual run-once priority, station order, output-usage limits, station delay and program-group postponement; API security coverage for Basic authentication, brute-force locking, throttled security events, role permissions, optional CSRF and audited state changes; settings-database persistence, legacy-value migration, new defaults, date conversion and fallback checks; and a complete plug-in install, activation, health, restart, update, rollback and disable lifecycle. The lifecycle test found and fixed a rollback defect where a plug-in whose new version failed to start had its previous files restored but remained disabled. Expected test failures are contained without noisy console output, and all tests continue to run against temporary data while OSPy is active. Updated the Czech translation.

Serialized option mutations with database writes to prevent a concurrent timer save from failing with `dictionary changed size during iteration`. Added a regression test for the locking behavior.

Errors block individual and bulk activation; warnings, checked-file counts and details are displayed in Plug-in management and Diagnostics and contribute to System status.
